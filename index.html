Webhooks

Notificações assíncronas de mudança de status. Configure sua URL HTTPS no Console — sem aprovação manual.

✅
Bradesco tenta a entrega até 5 vezes com backoff exponencial (1s → 2s → 4s → 8s → 16s). Após 5 falhas, o evento vai para a dead-letter queue e você recebe um alerta por e-mail.

Validação de assinatura HMAC-SHA256

Cada requisição inclui o header X-Bradesco-Signature: sha256=<hash>. Valide sempre antes de processar o evento.

// Express.js — validação de webhook Bradesco
const crypto = require('crypto');
const express = require('express');
const app = express();

// Use raw body para cálculo de HMAC
app.use(express.raw({ type: 'application/json' }));

function validateSignature(rawBody, signature, secret) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(rawBody)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expected, 'utf8'),
    Buffer.from(signature, 'utf8')
  );
}

import org.springframework.web.bind.annotation.*;
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.util.HexFormat;

@RestController
@RequestMapping("/webhooks")
public class BradescoWebhookController {

    @Value("${bradesco.webhook.secret}")
    private String webhookSecret;

    @PostMapping("/bradesco")
    public ResponseEntity<?> handleWebhook(
        @RequestBody byte[] rawBody,
        @RequestHeader("X-Bradesco-Signature") String signature
    ) {
        if (!isValidSignature(rawBody, signature)) {
            return ResponseEntity.status(401)
                .body(Map.of("error", "Assinatura inválida"));
        }

        WebhookEvent evento = objectMapper.readValue(rawBody, WebhookEvent.class);

        switch (evento.getEvento()) {
            case "desembolso.realizado" ->
                desembolsoService.processar(evento.getDados());
            case "proposta.assinada" ->
                propostaService.marcarAssinada(evento.getDados());
            case "proposta.cancelada" ->
                notificacaoService.notificarCancelamento(evento.getDados());
        }
        return ResponseEntity.ok(Map.of("received", true));
    }

    private boolean isValidSignature(byte[] body, String signature) {
        try {
            Mac mac = Mac.getInstance("HmacSHA256");
            mac.init(new SecretKeySpec(
                webhookSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"));
            String expected = "sha256=" +
                HexFormat.of().formatHex(mac.doFinal(body));
            // MessageDigest.isEqual é timing-safe
            return MessageDigest.isEqual(
                expected.getBytes(StandardCharsets.UTF_8),
                signature.getBytes(StandardCharsets.UTF_8));
        } catch (Exception e) { return false; }
    }
}

Débito automático de IPVA, licenciamento e multas direto na conta vinculada ao veículo. Consulta de débitos em tempo real e adesão ao débito automático — disponível em REST para integração tradicional e gRPC para cenários de altíssima frequência, como consulta no momento da venda do veículo.

✅
Integrado ao DETRAN-SP e RENAVAM nacional. Consulta de débitos em conformidade com a Resolução CONTRAN 809/2020. Disponibilidade 24/7, com SLA de 99,9%.
🏛️ Arquitetura Multi-Estado

Um contrato único. Quatro DETRANs hoje, qualquer número no futuro.

Cada DETRAN estadual tem suas próprias regras — pontuação de multas, descontos por antecipação, formato de parcelamento. Em vez de uma API por estado ou um contrato genérico que mente sobre o que é universal, o Débito Veicular usa um contrato-base estável + extensão tipada por UF. O endpoint nunca muda. O parâmetro uf decide qual adapter responde, e qual schema de extensão vem anexado.

Extensão por estado — 4 schemas hoje

Cada estado declara seu próprio formato dentro de extensaoEstadual. O campo tipo funciona como discriminador — o mesmo padrão usado em oneof no Protobuf e discriminator no OpenAPI 3.1.

  "maxParcelasIPVA": 6

SP é o único estado que expõe pontuação de multas na consulta — usado pela CNH por pontos. Parcelamento de IPVA em até 6x é regra estadual própria.

{
  "tipo": "RJ",
  "descontoAntecipacao": 0.03,
  "dataLimiteDesconto": "2026-02-28"
}

RJ concede 3% de desconto no IPVA para pagamento à vista até data-limite — informação que não existe no schema de nenhum outro estado.

extensaoEstadual quando uf=MG
{

  "isencaoFrotaAntiga": false,
  "anoFabricacaoLimiteIsencao": 1996

extensaoEstadual quando uf=BA

  "taxaLicenciamentoDigital": 45.20
}

BA cobra taxa específica para emissão digital do CRLV — não existe nos demais estados integrados.

⚠️
O contrato-base (debitos, valorTotal) nunca depende de extensaoEstadual. Um integrador que só precisa do valor total para cobrar o cliente nunca precisa olhar para o campo de extensão — funciona igual em qualquer dos 4 estados sem nenhum tratamento condicional.

Como adicionar um 5º estado — exemplo: Paraná (PR)

Esta é a prova de que o contrato aguenta crescer. Adicionar PR não toca em SP, RJ, MG ou BA — nem no contrato-base, nem no código de quem já integra hoje.

1
Definir o schema de extensão do PR — o DETRAN-PR oferece desconto progressivo por pagamento via PIX, algo que nenhum dos 4 estados atuais tem. O schema fica isolado: extensaoEstadual.tipo = "PR" com o campo descontoPixProgressivo.
2
Implementar o adapter PR no backend — uma função que traduz a resposta nativa do DETRAN-PR para o contrato-base + a extensão PR. Os adapters de SP/RJ/MG/BA continuam exatamente como estão — nenhum deploy neles.
3
Adicionar PR ao enum de uf na spec OpenAPI — é a única mudança no contrato público, e é estritamente aditiva (adicionar um valor a um enum nunca quebra clientes existentes).
4
Publicar o schema de extensaoEstadual.PR na documentação — como uma seção nova, sem reescrever a documentação dos outros 4 estados.
5
Nenhuma migração é necessária para parceiros que já integram SP/RJ/MG/BA — o contrato deles não muda em nenhum byte.

⚡ Nosso diferencial de plataforma

O primeiro banco brasileiro a oferecer gRPC nativo em API pública

A maioria dos bancos brasileiros expõe apenas REST — suficiente para a maior parte dos casos de uso, mas insuficiente para parceiros que operam em volume real. Uma concessionária verificando débitos de 200 veículos do pátio antes de uma feira, ou um marketplace validando milhares de anúncios simultaneamente, sente a diferença entre 120ms e 35ms por chamada multiplicada por escala. O Bradesco é a primeira instituição financeira do país a tratar isso como prioridade de produto, não como nota de rodapé técnica.

gRPC — perfil de uso

Definição do serviço — veicular.proto

Protocol Buffers v3
syntax = "proto3";

package bradesco.veicular.v1;

service VeicularService {
  // Unária — consulta simples de débitos
  rpc ConsultarDebitos(ConsultaRequest) returns (ConsultaResponse);

  // Server streaming — acompanha mudanças de status em tempo real
  rpc MonitorarDebitos(ConsultaRequest) returns (stream DebitoEvento);

  // Bidirectional streaming — consulta em lote para concessionárias/marketplaces
  rpc ConsultarLote(stream ConsultaRequest) returns (stream ConsultaResponse);

  rpc AderirDebitoAutomatico(AdesaoRequest) returns (AdesaoResponse);
}

message ConsultaRequest {
  string placa = 1;
  string renavam = 2;
}

message ConsultaResponse {
  string placa = 1;
  string renavam = 2;
  repeated Debito debitos = 3;
  double valor_total = 4;
}

message Debito {
  string tipo = 1;        // IPVA | LICENCIAMENTO | MULTA
  int32 exercicio = 2;
  double valor = 3;
  string vencimento = 4;   // ISO 8601
  StatusDebito status = 5;
}

enum StatusDebito {
  PENDENTE = 0;
  PAGO = 1;
  VENCIDO = 2;
}

message DebitoEvento {
  Debito debito = 1;
  string evento = 2;       // debito.pago | debito.vencido
}

message AdesaoRequest {
  string placa = 1;
  string conta_debito = 2;
}

message AdesaoResponse {
  string adesao_id = 1;
  bool confirmado = 2;
}

Início/Idempotência
🔁 Referência

Idempotência

Toda requisição de escrita (POST, PUT) em qualquer produto do Bradesco exige o header IdempotencyKey. Esta página é a referência única do padrão — válida para CaaS, PIX, Seguros, Consórcio e Débito Veicular.

✅
Por que existe: em caso de timeout ou erro de rede, o cliente não sabe se a operação foi processada no servidor. Reenviar a requisição com a mesma chave garante que o Bradesco nunca processe a mesma operação duas vezes — mesmo que a chamada HTTP original tenha sido perdida no caminho.

A regra em uma frase

Gere um UUID v4 por operação de negócio — não por tentativa HTTP. Reenvie a mesma chave em todas as tentativas de retry da mesma operação. Mude a chave apenas quando a operação for genuinamente nova.

Como o servidor se comporta

CenárioRespostaO que fazer
Chave novaProcessa normalmenteComportamento padrão
Mesma chave, mesmo payload200/201 — retorna o resultado original, sem reprocessarSeguro re-enviar em caso de timeout
Mesma chave, payload diferente409 IDEMPOTENCY_CONFLICTGerar uma chave nova — é uma operação diferente
Sem header IdempotencyKey400 BAD_REQUESTObrigatório em qualquer POST/PUT de escrita

Retry correto com backoff exponencial

Esta é a implementação de referência usada em todos os exemplos do portal (CaaS, PIX, Seguros, Consórcio, Débito Veicular). A chave é gerada uma única vez, antes do loop de tentativas — nunca dentro dele.

const { v4: uuidv4 } = require('uuid');

async function chamarComRetry(payload, maxTentativas = 4) {
  const idempotencyKey = uuidv4(); // gerada UMA vez, fora do loop
  let espera = 1000;
  for (let tentativa = 1; tentativa <= maxTentativas; tentativa++) {
    try {
      return await client.pix.criarCobranca(payload, { idempotencyKey });
    } catch (err) {
      if (err.status === 503 || err.status === 504) {
        await sleep(espera); espera *= 3; continue;
      }
      throw err; // outros erros não devem ser re-tentados
    }
  }
}

⚠️
Erro comum: gerar uma nova IdempotencyKey a cada tentativa de retry. Isso anula completamente a proteção — cada tentativa é tratada como uma operação nova, podendo causar processamento duplicado exatamente no cenário que a idempotência deveria prevenir.

Tempo de retenção

Chaves de idempotência são retidas por 24 horas em produção e 3 horas no Sandbox. Após esse período, a mesma chave pode ser reutilizada para uma nova operação sem conflito.

🔐Token expira em produção mesmo "funcionando há pouco"TOKEN_EXPIRADO401
Quando acontece: Em sistemas com tráfego intermitente — um job que roda a cada poucos minutos, por exemplo. O token foi gerado uma vez e ficou em uma variável global ou cache sem TTL.
Causa raiz real: O Bearer Token expira em 3600 segundos (1h). Qualquer processo de longa duração que gera o token uma única vez no startup vai falhar silenciosamente depois de 1h — o erro só aparece quando alguém tenta usar o sistema, não no momento em que o token expirou.
Solução — gerenciador de token com renovação proativa:
# Sintoma: 401 esporádico mesmo com token "válido" há poucos minutos
# Causa real: token gerado 1x e reusado por > 1h sem renovação
class TokenManager {
  constructor() { this.token = null; this.expiraEm = 0; }
  async getToken() {
    if (Date.now() < this.expiraEm - 60000) return this.token; // renova 60s antes
    const resp = await client.auth.gerarToken();
    this.token = resp.access_token;
    this.expiraEm = Date.now() + resp.expires_in * 1000;
    return this.token;
  }
}

💰Chave PIX que funcionava ontem, não encontrada hojeCHAVE_PIX_NAO_ENCONTRADA422
Quando acontece: Pagamento para uma chave que já recebeu PIX antes do seu sistema, mas falha agora.
Causa raiz real: O dono da chave a removeu ou portou para outra instituição no DICT do Bacen. Chaves PIX não são estáticas — o vínculo chave↔conta pode mudar a qualquer momento, fora do controle do Bradesco ou do pagador.
💡
Antes de tratar o erro como definitivo, ofereça ao usuário final a opção de inserir a chave novamente ou usar copia-e-cola — não assuma que é erro de digitação. Não há endpoint para "consultar dono da chave" por motivos de LGPD; a única forma de confirmar é tentar o pagamento.

🔁409 mesmo enviando "o mesmo payload" no retryIDEMPOTENCY_CONFLICT409
Quando acontece: Implementação de retry que deveria ser segura, mas recebe conflito.
Causa raiz real: O payload parece idêntico mas não é, byte a byte — geralmente porque um timestamp, UUID gerado dinamicamente, ou campo de metadado interno (como requestedAt) foi recalculado entre as duas tentativas. A comparação de idempotência é exata sobre o corpo serializado, não sobre a "intenção" da chamada.
⚠️
Monte o payload uma única vez e reenvie o mesmo objeto serializado em todas as tentativas — nunca reconstrua o corpo da requisição dentro do loop de retry. Veja a implementação de referência completa na página de Idempotência.
⏱️Timeouts concentrados em horários específicos do diaGATEWAY_TIMEOUT503/504
Quando acontece: Erros 503/504 que não acontecem em teste mas aparecem em produção, geralmente entre 9h–11h e 14h–16h.
Causa raiz real: Esses horários coincidem com o pico de volume do SPI (Sistema de Pagamentos Instantâneos do Bacen) — para operações PIX, a latência do upstream aumenta para todos os participantes do sistema, não só o Bradesco. Não é uma falha pontual da plataforma.
💡
Implemente retry com backoff exponencial (ver página de Idempotência) e, se seu volume é alto nesses horários, considere filas internas para distribuir a carga ao longo do tempo em vez de concentrar chamadas nos mesmos minutos.
📊Score insuficiente — cliente final pergunta "por quê"SCORE_INSUFICIENTE422
Quando acontece: Proposta reprovada por score, e o parceiro precisa explicar algo ao cliente final.
Causa raiz real: O cálculo de score considera variáveis de bureau de crédito externo (histórico de pagamento, dívidas ativas) que o Bradesco não controla nem pode detalhar — não é um valor numérico exposto pela API por política de crédito.
⚠️
A API nunca retorna o valor do score nem o motivo detalhado — apenas o resultado. Não construa UI que promete "mostrar o motivo da reprovação"; oriente o cliente final a consultar seu CPF/CNPJ diretamente nos birôs de crédito (Serasa, SPC) se quiser entender a composição do próprio score.
🪪CPF ou CNPJ "correto" retorna documento inválidoDOCUMENTO_INVALIDO400
Quando acontece: O número está certo (passa validação de dígito verificador local), mas a API rejeita.
Causa raiz real: A validação do Bradesco verifica mais que o dígito verificador — confirma que o documento está ativo na Receita Federal. CNPJs baixados, suspensos ou CPFs cancelados passam a validação matemática local mas são rejeitados pela API.
💡
Se seu sistema já valida o dígito verificador antes de chamar a API, esse erro específico geralmente significa que o documento tem problema cadastral real — não é bug de validação sua. Use o gerador do Sandbox para garantir documentos sintéticos sempre ativos durante testes.
🚦Rate limit atingido mesmo com volume abaixo do planoRATE_LIMIT_EXCEEDED429
Quando acontece: Plano contratado permite 50 req/s, mas o 429 aparece com volume médio bem menor que isso.
Causa raiz real: O limite é avaliado em janela deslizante de burst, não em média. Se 60 requisições chegam nos primeiros 2 segundos de uma janela de 60s — mesmo que o restante do minuto fique ocioso — o burst é excedido. O problema é concentração, não volume total.
💡
Distribua as chamadas no tempo (ex: fila com rate limiting client-side) em vez de disparar lotes inteiros de uma vez. Veja os limites de burst exatos na página de Rate Limits.

Validação de assinatura

O Bradesco inclui o header x-bradesco-signature com HMAC-SHA256 do payload. Valide sempre antes de processar.

// Express.js — validação de webhook PIX
const crypto = require('crypto');
app.use(express.raw({ type: '*/*' }));

app.post('/webhooks/pix', (req, res) => {
  const sig = req.headers['x-bradesco-signature'];
  const expected = 'sha256=' + crypto
    .createHmac('sha256', process.env.PIX_WEBHOOK_SECRET)
    .update(req.body).digest('hex');

  if (!crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(sig)))
    return res.status(401).end();

  const { pix } = JSON.parse(req.body);
  pix.forEach(p => {
    console.log(`PIX recebido: R$ ${p.valor} de ${p.pagador.nome}`);
    // processar liquidação do pedido p.txid...
  });
  res.status(200).end();
});
# FastAPI — validação webhook PIX
import hmac, hashlib, os
from fastapi import FastAPI, Request, HTTPException

app = FastAPI()

@app.post("/webhooks/pix")
async def pix_webhook(request: Request):
    raw = await request.body()
    sig = request.headers.get("x-bradesco-signature", "")
    secret = os.environ["PIX_WEBHOOK_SECRET"]
    expected = "sha256=" + hmac.new(
        secret.encode(), raw, hashlib.sha256).hexdigest()
    if not hmac.compare_digest(expected, sig):
        raise HTTPException(status_code=401)
    data = await request.json()
    for pix in data["pix"]:
        print(f"PIX: R$ {pix['valor']} de {pix['pagador']['nome']}")
    return {"ok": True}

@PostMapping("/webhooks/pix")
public ResponseEntity<?> pixWebhook(
    @RequestBody byte[] body,
    @RequestHeader("x-bradesco-signature") String sig) {
  Mac mac = Mac.getInstance("HmacSHA256");
  mac.init(new SecretKeySpec(
    webhookSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"));
  String expected = "sha256=" +
    HexFormat.of().formatHex(mac.doFinal(body));
  if (!MessageDigest.isEqual(expected.getBytes(), sig.getBytes()))
    return ResponseEntity.status(401).build();
  PixWebhookPayload payload = mapper.readValue(body, PixWebhookPayload.class);
